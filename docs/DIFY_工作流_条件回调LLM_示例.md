# Dify 工作流示例：条件判断 → HTTP 回调本后端 → LLM 回答

以应用「11」（或 test1）为例，在 Dify 里按步骤拖节点即可实现：**先判断是否需要查工作台，需要则请求本零售后端，再把结果交给 LLM 生成回答**。

---

## 一、整体结构（节点顺序）

```
[开始] → [条件分支：是否需要回调] 
              ├─ 是 → [HTTP 请求：调本后端工作台] → [LLM：基于回调结果+用户问题回答] → [结束]
              └─ 否 → [LLM：仅根据用户问题回答] → [结束]
```

两路最后可汇合到一个「回复」节点，或各自接一个 LLM 后分别结束（看 Dify 版本是否支持多分支汇合）。

---

## 二、变量名约定（方便后面引用）

| 变量名 | 含义 | 来源 |
|--------|------|------|
| `query` | 用户本轮输入 | 对话型应用默认传入 |
| `name` / `name2` | 提示词变量（你已有） | 应用变量 |
| `need_callback` | 是否需要回调工作台 | 条件节点或 LLM 判断输出 |
| `workbench_result` | 工作台接口返回的 JSON 字符串 | HTTP 请求节点输出 |

---

## 三、逐步操作（在应用「11」的工作流编排里）

### 步骤 1：确认「开始」与输入

- 打开应用「11」→ 编排（工作流）。
- 默认已有 **开始** 节点，输入变量里会有 **query**（用户问题）。若你之前配了 `name`、`name2`，也在这里。
- 无需改开始节点，记下 **query** 即可。

### 步骤 2：加「条件分支」节点（第一个判断）

- 从左侧拖入 **「条件分支」**（或「IF/ELSE」），接到「开始」之后。
- 配置分支条件，二选一即可：
  - **方式 A（推荐）**：用 **「LLM 判断」** 节点代替条件分支：  
    - 拖入「LLM 判断」，输入为 `{{query}}`，让模型输出是否需要查工作台（例如输出 `yes`/`no` 或 `true`/`false`）。  
    - 把输出变量命名为 `need_callback`（或你喜欢的名字），后面分支根据该变量判断。
  - **方式 B**：用 **「条件分支」** 的「关键词」规则：  
    - 若 `query` 包含「工作台」「订单」「库存」「销量」「待发货」等 → 走「需要回调」分支；否则走「不需要」。
- **输出**：  
  - 需要回调的一路：接步骤 3。  
  - 不需要的一路：接步骤 4 的「仅 LLM」那条。

### 步骤 3：加「HTTP 请求」节点（回调本后端）

- 在「需要回调」的那条线上，拖入 **「HTTP 请求」**。
- 配置：
  - **URL**：`GET http://host.docker.internal:8080/api/store/workbench`  
    - Dify 与后端同机时用 `host.docker.internal`；若 Dify 在别的机器，改为后端实际地址，如 `http://192.168.x.x:8080/api/store/workbench`。
  - **Method**：GET。
  - 若你后端对 `/api/store/*` 做了登录鉴权：在 **Headers** 里加一项（后端需支持内部 Key 校验）：  
    - Key：`X-Internal-Api-Key`  
    - Value：你在后端配置的 `agent.dify-internal-key`（需先在 application.yml 和代码里加上该校验，见 [DIFY_INTEGRATION.md](./DIFY_INTEGRATION.md)）。
- **输出变量**：把 HTTP 响应体映射到变量 **workbench_result**（或选「响应 body 文本」并命名为 `workbench_result`）。若 Dify 支持「JSON 路径」，可只取其中一段再赋给 `workbench_result`。

### 步骤 4：加「LLM」节点（生成回答）

需要两条 LLM 分支（或一个 LLM 用不同输入）：

**4.1 需要回调的那条（接在 HTTP 节点后）**

- 拖入 **「LLM」** 节点，接在「HTTP 请求」之后。
- **系统提示**（示例）：
  ```text
  你是店家工作台助手。下面是你从系统获取到的最新工作台数据（JSON），请根据用户问题用简短中文回答，不要编造数字。
  工作台数据：{{workbench_result}}
  ```
- **用户输入 / 问题**：`{{query}}`（或「用户问题：{{query}}」）。
- 该 LLM 节点的输出即「基于回调结果的回答」，接到 **结束** 或 **回复** 节点。

**4.2 不需要回调的那条（直接回答）**

- 再拖一个 **「LLM」** 节点，接在「条件分支」的「否」分支后。
- **系统提示**：按你原来的助手人设即可（例如：你是店家助手，仅根据用户问题回答，不涉及工作台数据时不要瞎编数据）。
- **用户输入**：`{{query}}`。
- 输出接到 **结束** 或 **回复** 节点。

### 步骤 5：结束 / 回复

- 若 Dify 版本支持「回复」节点：把两个 LLM 的输出都连到同一个「回复」节点，作为对话的最终 answer。
- 若不支持多分支进同一节点：可保留两路各自结束，只要每条分支最后都有一个 LLM 输出作为该分支的 answer 即可（对话型应用会取工作流终点输出作为 answer）。

---

## 四、变量名汇总（复制到 Dify 里用）

- 用户问题：`{{query}}`
- 是否需要回调：`{{need_callback}}`（若用 LLM 判断）
- 工作台数据：`{{workbench_result}}`（HTTP 节点输出）
- 提示词变量（你已有）：`{{name}}`、`{{name2}}`

---

## 五、与本项目后端的对应关系

- 后端调 Dify：`POST {agent.api-url}/chat-messages`，body 里带 `query`、`user`、`conversation_id`、`inputs`（含 name、name2）。
- Dify 工作流里「回调」的是本后端：`GET /api/store/workbench`（工作台汇总）。若要流量数据，可再加一个 HTTP 节点：`GET /api/store/traffic?range=7`，输出变量如 `traffic_result`，在 LLM 提示里同样用 `{{traffic_result}}` 引用。

按上述顺序：**条件（或 LLM 判断）→ HTTP 回调 → LLM** 即可在 Dify 内完成「按需回调、再让 LLM 回答」；后端无需改，仍是一次 `chat-messages` 调用，超时与异常处理已由后端 60 秒超时和异常提示覆盖。
