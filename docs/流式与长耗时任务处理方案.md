# 流式输出与长耗时任务处理方案

本文档说明：**智能助手 / Dify API** 场景下，流式输出与长耗时任务在**本项目当前做法**与**企业级常见做法**的对比与演进建议。

---

## 一、问题与目标

| 场景 | 问题 | 目标 |
|------|------|------|
| **流式输出** | 用户希望「边生成边看到字」，而不是等整段话一次性返回 | 后端以 SSE/流式方式把 Dify 或 LLM 的逐字/逐块结果推给前端，前端做打字机效果 |
| **长耗时任务** | 工作流或 LLM 推理超过几十秒，易遇 HTTP 超时、网关超时，用户长时间白屏 | 要么拉长超时 + 流式减轻体感，要么改为「提交即返回 + 轮询/推送取结果」，保证不超时、可追溯 |

---

## 二、本项目当前做法

### 2.1 架构概览

- **入口接口**：`POST /api/store/agent/chat`（`StoreAgentController.chat`）  
  - 入参：`message`（必填，最多 20 字）、`conversation_id`（可选，多轮对话用）。  
  - 出参：`reply`（助手回答）、`conversation_id`（多轮时返回）。  
- **业务 Service**：`StoreAgentService.chat(userId, message, conversationId)`  
  - 负责限流、路由到不同 Provider（OpenAI 兼容 / Dify）、拼接 RAG 上下文、落会话与错误日志。  
- **OpenAI 兼容模式（当前默认）**（`agent.provider=openai`）  
  - 构造 `system` + `user` 消息；`system` 中注入工作台助手指令与 `ragService.retrieve` 返回的知识块。  
  - 调用 `{agent.api-url}/chat/completions`（例如 Zeabur 提供的 OpenAI 兼容接口），等待一次性 JSON 回复。  
  - 支持工具调用：`get_workbench_summary`、`get_traffic_summary`，按 OpenAI 工具调用规范在 Java 侧执行后二轮调用模型。  
- **Dify 模式**（`agent.provider=dify`）  
  - 通过 `StoreAgentService.chatWithDifyAndPersist` → `chatWithDify` 调用 `POST {agent.api-url}/chat-messages`，`response_mode=\"blocking\"`，一次性拿到 `answer`。  
  - Dify 的 `conversation_id` 会与本项目的 `agent_conversation` 记录关联，用于后续续聊；当前 `response_mode` 仍为 blocking，**无流式**。

### 2.2 长耗时与超时

| 项 | 实现 |
|----|------|
| **HTTP 超时** | 使用 `SimpleClientHttpRequestFactory` 创建 `RestTemplate`，`connectTimeout` 与 `readTimeout` 来自配置 `agent.dify-timeout-seconds`（默认 60 秒），统一限制对 Dify/LLM 的最大等待时间。 |
| **限流** | Redis：以 `agent:ratelimit:{userId}` 作为 Key，在调用前使用 `setIfAbsent` + 过期时间 30 秒，实现「每个用户 30 秒内仅能调用一次助手」的频控，避免长耗时任务被重复触发与堆积。 |
| **错误与日志** | 所有调用异常（如超时、401、网络错误）会记录到 `agent_error_log` 表，字段包含用户、会话、异常栈、发生时间等；对外统一返回固定文案「服务暂时不可用，请稍后再试。」，不把内部错误信息透出给前端。 |

### 2.3 会话与持久化

- **会话入口**：前端多轮对话通过传入 `conversation_id`（本项目的会话 ID），后端在 `StoreAgentService.chat` 中根据 `conversation_id` 查找对应 `AgentConversation` 记录。  
- **消息持久化**：每次对话的 user/assistant 消息落到 `agent_message` 表中，包含 role、content、token 预估、时间戳等，便于回放和审计。  
- **Dify 模式上下文**：当使用 Dify 模式时，会话表中额外保存 `dify_conversation_id`，Java 根据配置的 `conversation_max_messages`（默认 20）加载最近 N 条历史，拼接为「历史摘要 + 最近对话 + 当前问题」形式的 query，或直接透传 `dify_conversation_id` 给 Dify 用于续聊。  
- **OpenAI 模式上下文**：当前对话上下文主要依赖本轮问题 + `ragService` 的知识检索结果，不做长会话上下文拼接。  
- **流式**：当前 **未实现** 流式接口，无论 Dify 还是 OpenAI 模式，前端都只能收到一次性完整 `reply`。

### 2.4 小结（本项目现状）

- **流式**：未做，均为同步阻塞返回。
- **长耗时**：通过 **可配置读超时**（默认 60s）和 **限流** 控制；超时或异常后统一写错误表并返回友好提示。若 Dify 工作流超过 60 秒，会触发读超时。
- **异步 / MQ**：未使用；所有调用均为同步等待 Dify/LLM 响应后返回。

## 三、本项目可选演进方案

### 3.1 流式输出（推荐优先）

**目标**：用户发问后，前端以打字机效果逐字/逐块展示回复。

**做法**：

1. **后端**
   - 新增流式接口，例如 `GET /api/store/agent/chat/stream?message=xxx&conversation_id=xxx`，或沿用 POST 但返回 `Content-Type: text/event-stream`。
   - Dify 调用时改为 `response_mode: "streaming"`，使用可消费 SSE 的客户端（如 `WebClient` + `Flux<ServerSentEvent>` 或按行读 `InputStream`）接收 Dify 的 `text/event-stream`。
   - 将 Dify 的 SSE 事件（如 `message` 的 delta、或 `workflow_finished` 的 answer）通过 **Spring `SseEmitter`** 或 **`ResponseBodyEmitter`** 推给前端；设置合理超时（如 2～5 分钟）和异常时 `completeWithError`。

2. **前端**
   - 使用 `EventSource(url)` 或 `fetch` + `ReadableStream` 消费 SSE，每收到一段就追加到 UI。

3. **配置**
   - 流式读超时建议单独配置（如 `agent.dify-stream-timeout-seconds`），大于 blocking 的 60s，避免长回答被中途断开。

**不必须引入 MQ**：流式是「长连接 + 边收边推」，同步即可完成。

### 3.2 长耗时仅「拉长超时 + 流式」

- 若 Dify 工作流多在 1～2 分钟内完成：将 **blocking 读超时** 调大（如 120s），和 Nginx/网关超时一起调；同时 **上线流式**，用户先看到部分输出，体感更好。
- 仍不必须异步或 MQ。

### 3.3 长耗时「异步 + 轮询」（可选）

**适用**：工作流经常超过 2 分钟，或网关/链路超时无法再放大。

**做法**：

1. 后端提交 Dify 请求时，若 Dify 支持「异步创建消息」并返回 `message_id`，则只做「提交」，拿到 `message_id` 后立即返回前端（如 `{ "task_id": "xxx", "status": "pending" }`）。
2. 结果获取二选一：
   - **轮询**：前端定时调后端「查询任务结果」接口，后端根据 `message_id` 调 Dify 的「获取消息结果」API，或从 Redis/DB 读已写入的结果；后端也可先查 Redis 再查 Dify，减少对 Dify 的重复调用。
   - **Webhook**：若 Dify 支持完成回调，后端接收 Webhook 后将结果写入 DB 或 Redis（key 如 `agent:result:{message_id}`，TTL 10 分钟），前端仍通过「查询任务结果」轮询你们自己的接口。
3. **可选 MQ**：Webhook 收到后发 MQ，由消费者写库/写 Redis，解耦与削峰；非必须，直接写库/写 Redis 也可。

### 3.4 缓存（可选）

- **结果缓存**：同一用户、同一问题（如 query 的 hash）在短时间（如 5～10 分钟）内可命中 Redis，key 如 `agent:cache:{userId}:{queryHash}`，直接返回缓存回复，降低对 Dify 的调用与延迟。注意多轮对话的连贯性，仅适合「单轮相同问句」的降成本。
- **会话/上下文**：将「最近 N 条消息」或摘要缓存到 Redis（如 `agent:context:{userId}`），减少对 DB 的频繁查询，与现有 `conversation_max_messages` 策略一致。

## 四、企业级常见做法

### 4.1 流式输出

| 做法 | 说明 |
|------|------|
| **SSE 透传** | 网关/后端对 LLM 或 Dify 使用 `response_mode=streaming`，将收到的 `text/event-stream` 按 chunk 透传给前端，前端用 `EventSource` 或 `fetch` + stream 消费。 |
| **统一事件格式** | 后端解析 LLM/Dify 的 SSE 事件，转换为统一内部格式（如 `{ "type": "delta", "content": "..." }`），再通过 SSE 或 WebSocket 推给前端，便于多模型/多供应商切换。 |
| **超时与断开** | 服务端对 SSE 连接设置最大存活时间（如 5 分钟）和「用户取消」支持（前端关闭连接或发取消请求）；后端收到取消后尽快关闭对上游的请求并释放资源。 |
| **限流与背压** | 流式长连接占用线程/连接资源，企业级通常配合连接数限流、按用户/租户的并发限制，避免单用户占满资源。 |

### 4.2 长耗时任务

| 做法 | 说明 |
|------|------|
| **同步 + 足够超时** | 对「可接受几十秒」的场景，拉长 HTTP 读超时与网关超时，并配合流式减轻体感；实现简单，运维需统一配置网关与上游超时。 |
| **异步提交 + 轮询** | 提交请求后立即返回 `task_id`，前端轮询「任务状态/结果」接口；后端根据 `task_id` 查 DB 或 Redis（由 Worker 或 Webhook 写入），避免长连接与网关超时。 |
| **异步 + 推送** | 任务完成后通过 WebSocket 或 SSE 推送给对应用户，前端无需轮询；需维护用户与连接的关系（如 userId/sessionId -> WebSocket）。 |
| **MQ 解耦** | 提交请求时写入「待处理」状态并投递 MQ，Worker 消费后调 Dify/LLM，将结果写 DB 或 Redis，再通过 Webhook/回调或推送通知前端；便于削峰、重试与水平扩展。 |
| **超时与重试** | 对 Worker 调用的上游设置超时与重试策略；任务表记录状态（pending / running / completed / failed），便于排查与补偿。 |

### 4.3 缓存与稳定性

| 做法 | 说明 |
|------|------|
| **多级缓存** | 本地缓存（如 Caffeine）+ Redis，热点会话或结果先走本地再走 Redis，最后才调 Dify，降低延迟与成本。 |
| **缓存 Key 规范** | 按租户/用户/场景设计 Key（如 `agent:result:{tenantId}:{userId}:{queryHash}`），统一 TTL 与失效策略，避免误用旧答案。 |
| **降级** | 当 Dify 或 LLM 不可用/超时时，返回兜底话术或从缓存返回上一次结果（若业务允许），并记录降级指标。 |
| **可观测** | 对超时、失败率、流式首包延迟、端到端延迟打点，便于做 SLA 与容量规划。 |

### 4.4 安全与合规

| 做法 | 说明 |
|------|------|
| **鉴权** | 流式/轮询接口与同步接口一致，校验 JWT 或会话，确保仅本人可拉取自己的对话与结果。 |
| **敏感信息** | 日志与错误表中不记录完整 question/answer，仅记录脱敏或 ID，符合企业审计与隐私要求。 |
| **速率与配额** | 按用户/租户限流（如 Redis 计数），并可选按日/月配额控制调用量，防止滥用。 |

## 五、对照小结

| 维度 | 本项目当前 | 本项目可选演进 | 企业级常见 |
|------|------------|----------------|------------|
| **流式** | 无，blocking 一次性返回 | 增加 SSE 接口，Dify 用 streaming，SseEmitter 推前端 | SSE/WS 透传或统一事件格式，限流与超时 |
| **长耗时** | 60s 读超时，超时后错误表 + 固定提示 | 拉长超时 + 流式；或异步 + 轮询（+ 可选 MQ） | 同步大超时 + 流式，或异步 + 轮询/推送 + MQ |
| **异步/MQ** | 未使用 | 长耗时可选：Webhook + Redis/DB，或 MQ 解耦 | 常见用 MQ 解耦提交与执行，结果写库/缓存 |
| **缓存** | 仅限流 Redis | 可选结果缓存、会话上下文缓存 | 多级缓存、Key 规范、降级与可观测 |
| **会话** | Java 持久化（conversation/message） | 可加 Redis 缓存最近上下文 | 会话与上下文分层存储，缓存 + DB |
