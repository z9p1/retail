# Dify 企业级回调鉴权：不用店家 JWT 的常见做法

后端接口需要鉴权时，Dify 工作流里的 HTTP 节点**不能带用户 JWT**（Dify 拿不到、也不该存店家 token）。企业里常见三种方式：

---

## 一、内部 API Key（服务间密钥）

**做法**：后端为「被 Dify 调用的接口」单独设一套鉴权：请求头带 **X-Internal-Api-Key**（或类似），且与配置的密钥一致则放行，**不校验 JWT**。

- **优点**：简单、不依赖用户态；Dify 工作流里 HTTP 节点固定配一个 Key 即可。
- **缺点**：回调拿不到「当前是哪个店家」，适合**单租户**或「全店汇总」类接口（如工作台汇总、流量汇总）。
- **适用**：工作台、流量、库存汇总等「不区分登录用户」的只读查询。

本项目已支持：配置 `agent.dify-internal-key` 后，对 `/api/store/workbench`、`/api/traffic` 等带该 Key 的请求放行。

---

## 二、一次性回调 Token（带用户上下文）

**做法**：用户发消息时，后端生成**短期 token**（如 60 秒有效、Redis 存 userId/conversationId），把 token 通过 Dify 的 **inputs** 或 **query** 传给工作流；Dify 的 HTTP 节点回调时在 Header 或 Body 里带上该 token；后端校验 token 后取出 userId，再按该用户/店家查数据。

- **优点**：回调能关联到「当前对话的店家」，可做多租户、按店隔离。
- **缺点**：需在请求 Dify 时注入 token、工作流里把 token 传给 HTTP 节点，实现稍复杂。
- **适用**：多租户、按店家查订单/库存等。

---

## 三、专用「内部」接口 + 内部 Key

**做法**：不放开现有 `/api/store/*`，而是单独提供 **/api/internal/workbench** 等，只校验内部 Key；若需按用户过滤，在 Body 里传 `userId` 或 `storeId`（由调用方从上下文带过来）。

- **优点**：与对用户暴露的接口隔离，权限清晰。
- **缺点**：多一套 URL，需在工作流里拼 Body。

---

## 小结

| 方式           | 鉴权方式           | 能否带用户/店家 | 典型场景       |
|----------------|--------------------|------------------|----------------|
| 内部 API Key  | X-Internal-Api-Key | 否，全店/全局    | 工作台、流量汇总 |
| 一次性 Token  | 短期 token 回传    | 是               | 多租户、按店查   |
| 专用内部接口  | 内部 Key + 可选 Body | 可传 storeId 等  | 隔离、审计      |

**常见选择**：只查「全店汇总」用 **内部 Key**；要按店家查用 **一次性 Token** 或 **专用接口 + Body 传 storeId**。本项目先实现内部 Key 放行，满足 Dify 回调工作台/流量等汇总接口。
